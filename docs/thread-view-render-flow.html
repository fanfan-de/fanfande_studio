ThreadView 数据到渲染流程图

这张图把 ThreadView 看成一条流水线：上游是会话数据，中间把 turns 拆成可渲染的 rows，最后按 row 和 trace item 类型落到具体 UI 组件。

数据 归一化与状态 布局容器 渲染分发 屏幕结果
1. 数据输入 props
activeTurns

主会话的用户 turn 与 assistant turn。

activeSession

当前 session、运行时状态、模型选择等上下文。

pendingPermissionRequests

阻塞当前 session 的权限请求。

2. 行归一化 rows
buildThreadDisplayRows()

把 turns 拆成 UI row，不再简单一 turn 对一块 UI。

process rows

reasoning/tool 可被拆成 header 与 item。

message rows

保留用户消息与最终 assistant 回合。

displayRows

后续渲染只面向 row，而不是直接遍历 turns。

3. Thread 状态 behavior
滚动状态

锁底、恢复位置、用户向上阅读时保持 detached。

虚拟列表

长 thread 时只渲染可见窗口附近的 rows。

轻量模式

侧栏 resize 时降低昂贵渲染。

4. 布局外壳 DOM
VisibleThreadView

真正可见的 ThreadView 分支。

section.thread-shell

thread 区域外壳，普通模式下透明。

div.thread-column

独立滚动列，所有 rows 都进入这里。

5. 行级分发 row.kind
renderDisplayRow(row)

根据 row.kind 选择用户消息、权限请求、process item 或 assistant turn。

UserTurnArticle

用户输入气泡。

assistant-turn

assistant 回合主体。

PermissionRequestInlinePrompt

权限请求作为 thread 内 row 插入。

6. 屏幕结果 UI
最终回复正文

response section 优先，以轻量正文形式呈现。

执行轨迹

reasoning、tools、workflow 等被折叠或弱化显示。

消息动作

copy、branch、side chat、fork 贴近对应 assistant 回复。

Assistant turn 内部再分一层

AssistantTurnSections

把 items 按 section 分组。

AssistantTraceBlockView

渲染一个 section block。

TraceItemView

单条 trace item 的统一入口。

traceItemRenderers

按 kind 落到 text/tool/image/patch 等 UI。

读代码顺序

  1. WorkbenchPaneSurface：确认 ThreadView 被放在哪。
  2. ThreadView：入口与可见/不可见分支。
  3. buildThreadDisplayRows：turns 如何变成 rows。
  4. renderDisplayRow：row 如何变成组件。
  5. AssistantTurnSections：assistant items 如何分 section。
  6. TraceItemView：trace item 如何落到具体 renderer。

深入 1：数据输入是什么形状

ThreadView 拿到的不是一整块 HTML，而是一组会话对象。最核心的是 activeTurns：用户输入和 assistant 输出按时间排成数组。assistant turn 里面再通过 items 保存推理、工具调用、文件变更、最终回复等细粒度内容。

深入 2：行归一化的逻辑

buildThreadDisplayRows() 的作用是把“会话数据”转换成“屏幕行”。它不直接渲染 UI， 而是先生成 ThreadDisplayRow[]，让后面的虚拟滚动、折叠展开、row 级分发都面对同一种结构。

1 没有 session 就返回空

如果 activeSession 不存在，ThreadView 没有可渲染上下文，直接返回 []。

2 可选插入 session banner

只在 showSessionBanner 和 readOnlySideChat 同时成立时插入。

3 逐个扫描 activeTurns

user turn 通常变成 user-turn；assistant turn 会进入更复杂的拆分逻辑。

4 assistant 拆成过程 + 正文

必要时先产出 process-header 和 process-item，最后产出 assistant row。

5 最后追加权限请求

如果有待处理权限且当前不在 resolving，就追加一个 permission-request row。

归一化前：turns 是按对话顺序排的

它更像“消息历史”，还不适合直接做虚拟列表和行级渲染。

activeTurns = [

      { id: "response-1", kind: "text", text: "ThreadView 分为..." }

归一化中：assistant turn 会被拆开

过程 trace 和最终回复分离，折叠逻辑也在这里决定。

for (const turn of activeTurns) {
  if (turn.kind === "user") {
    rows.push({
      kind: "user-turn",
      rowID: turn.id,
      estimatedHeight: estimateUserThreadRowHeight(turn),
      turn
    });
    continue;
  }

  const processBlocks = buildAssistantTraceDisplayBlocks(turn.items);
  const expanded = readProcessTraceExpanded(turn.id);

  if (processBlocks.shouldRenderProcessTrace) {
    rows.push({
      kind: "process-header",
      rowID: `process-header:${turn.id}`,
      expanded
    });

    if (expanded) {
      rows.push({
        kind: "process-item",
        rowID: "process-item:assistant-1:0:reasoning-1:0",
        item: reasoningItem,
        section: "reasoning"
      });
    }
  }

  rows.push({
    kind: "assistant",
    rowID: turn.id,
    renderedItems: responseItems,
    processPrefixItems,
    insertedUserTurns
  });
}

这里最容易混淆的地方
  • process-header 只是折叠入口，不是最终回复；它显示 “Processed / reasoning / tools” 那一类过程摘要。
  • process-item 来自 assistant trace block，只有过程区域展开或正在折叠动画中才会逐条插入。
  • assistant row 仍然代表这个 assistant turn 的主体，通常承载最终回复、动作按钮、diff card、inline side chat。
  • estimatedHeight 是虚拟滚动的预估高度，真实渲染后还会被测量修正。

深入 3：buildAssistantTraceDisplayBlocks 的逻辑

这个方法专门处理 assistant turn 内部的 trace items。它的目标是判断哪些内容应该放进 Processed 折叠区，哪些内容应该留在 assistant 正文区。可以把它理解成： 先把 item 分组，再找到最终回复的位置，然后用最终回复作为分界线切开。

1 过滤不可见 item

先跑 filterRenderedAssistantTraceItems，按 traceVisibility 和 showFileChanges 去掉不该显示的项。

2 按 section 分 block

buildAssistantTraceBlocks 把连续同类 section 合并成 block，例如 reasoning、tools、response。

3 寻找最终 response

findFinalResponseBlockIndex 从后往前找最后一个包含正文 text 的 response block。

4 切出 process 候选

最终 response 之前的 blocks，加上前序 assistant turn 的 processPrefixItems，组成过程区候选。

5 决定是否单独渲染

如果只是很短的一条 reasoning，就不拆 Processed 区；否则生成 processBlocks 和 mainBlocks。

输入：trace items 还只是扁平数组

每个 item 可能显式带 section，否则会按 kind 推断默认 section。

items = [
  {

    kind: "reasoning",

    kind: "patch",

    ]
  },
  {
    id: "a1",
    kind: "text",
    text: "ThreadView 的渲染流程可以这样理解..."
  }
];

同类 section 会合并；file-change 比较特殊，会被集中到一个 block。

输出：切成 processBlocks 和 mainBlocks

最终 response 之前的过程内容进 Processed；最终 response 及之后留在正文区。

  { sectionKey: "reasoning", items: [r1] },
  { sectionKey: "tools", items: [t1] }

mainBlocks = [
  { sectionKey: "response", items: [a1] },

return {
  blocks,
  processBlocks,
  mainBlocks,
  shouldRenderProcessTrace: true
};
关键判断规则
  • shouldCollapseReasoningAndTools 为 false 时，不会寻找 process 分界，所有 block 都留在 mainBlocks。
  • 只有找到“最后一个有 text 正文的 response block”，并且它前面确实有过程内容时，才可能生成 processBlocks。
  • processPrefixItems 用来把同一轮 assistant run 里更早 assistant turn 的过程 item 合并到最终回复前面。
  • 单条很短的 reasoning 会被内联显示，避免页面上出现一个只有一句话的 Processed 折叠区。
  • file-change block 会汇总文件变更项，渲染时还会通过 summarizeFileChangeItems 做进一步压缩。

深入 4：流式输出时 activeTurns 一直变化怎么办

ThreadView 不直接维护“正在输出的字符串”。它把 activeTurns 当成外部传入的实时快照： 每来一段流式事件，上层状态更新 activeTurns，ThreadView 重新计算 displayRows，React 再用稳定的 turn.id / rowID 更新同一块 UI。

1 上层收到流式事件

assistant turn 的 items、runtime.phase、isStreaming 被更新。

2 ThreadView 重新 render

displayRows 依赖 activeTurns，所以每次快照变化都会重新归一化。

3 流式中先不切 Processed

canCollapseAssistantProcessTrace 在 streaming 时为 false，过程和正文先留在 assistant row 里。

4 同一 row 持续长高

rowID 通常仍是 assistant turn id，React 更新内容，ResizeObserver 重新测量高度。

5 完成后重新分块

isStreaming 变 false 后，reasoning/tools 才可能被折到 Processed 区。

这时会显示 ephemeral hint，例如 Thinking、Running tools。

流式输出的几个关键点
  • activeTurns 是快照，不是 append-only 日志；每次变化都会重新跑 buildThreadDisplayRows()。
  • 流式中 showFileChanges 为 false，文件变更类内容通常等 assistant 完成后再进入渲染。
  • 当最新 assistant 仍是同一个 id 且还在 streaming，滚动同步会使用 smooth follow，让视图跟随正在增长的响应。
  • 如果用户向上滚动，scrollModeRef 会变成 detached，后续流式更新不会强制把用户拉回底部。
  • 完成瞬间 isStreaming: true → false 会触发一次完成态处理：保留当前位置，并允许过程区自动折叠。
  • 虚拟列表通过 rowID 缓存测量高度；ResizeObserver / MutationObserver 会在内容增长后修正布局。

深入 5：以 displayRows 为中心看前后数据处理

displayRows 是 ThreadView 最重要的中间层。它把“业务数据形状” 转成“屏幕行形状”：前面接 activeTurns、权限请求、trace 可见性、折叠状态； 后面接虚拟滚动、滚动跟随、renderDisplayRow(row) 和具体 UI 组件。

前 WorkbenchPaneSurface

通过 useConversationTurns 取出当前 session 的 activeTurns，再把 session、权限、side chat 等状态传给 ThreadView。

入 VisibleThreadView props + UI state

把外部 props 和内部折叠状态合在一起，得到 buildThreadDisplayRows 的输入。

中 buildThreadDisplayRows

遍历 turns，把 user、assistant、process trace、permission request 归一化为 row。

出 ThreadDisplayRow[]

每个 row 都有稳定 rowID 和 estimatedHeight，适合虚拟列表和渲染分发。

后 renderThreadRows

短列表直接 map；长列表先算虚拟 range，再只渲染可见 rows。

displayRows 之前：输入不是只有 activeTurns

它还依赖权限请求、trace 可见性、side chat、过程区折叠状态。

displayRows 之后：虚拟滚动和 row 渲染

后续代码不再直接遍历 turns，而是消费统一 row 队列。

const shouldVirtualizeThreadRows =
  displayRows.length >= 80;

const threadVirtualLayout =
  buildThreadVirtualLayout(displayRows, heightCache);

const threadVirtualRange = shouldVirtualizeThreadRows
  ? findThreadVirtualRange(threadVirtualLayout, viewport)
  : { items: threadVirtualLayout.items };

function renderThreadRows() {
  if (!shouldVirtualizeThreadRows) {
    return displayRows.map(renderDisplayRow);
  }

  return threadVirtualRange.items.map((item) =>
    renderDisplayRow(item.row)
  );
}
先建立一个心智模型
  • displayRows 不是消息数组，而是“屏幕施工队列”：每个 row 都告诉 UI 这里要放一块什么。
  • user-turn 和 assistant 是消息主体；process-header 和 process-item 是从 assistant 主体里拆出来的过程区。
  • permission-request 不在 activeTurns 里，但为了让它看起来像 thread 的一部分，也会被包装成 row。
  • session-banner 是一个特殊兼容分支：它不表示“主 ThreadView 内嵌了 side chat 窗口”，只表示“当前 ThreadView 自己正在渲染一个 side-chat session”。如果产品只把 side chat 放到右侧栏，这一行基本不会出现。
row.kind 应该怎么理解 什么时候产生 / 消失 关键字段怎么读 后续渲染入口
session-banner 把它当成“standalone side-chat session 的提示条”，不是主 thread 里的 side chat 面板。它只说明当前 session 本身是 kind: "side-chat"。 只在 showSessionBanner 和 readOnlySideChat 同时成立时插到最前面。当前右侧栏 / inline 的 InlineSideChatThread 会传 showSessionBanner={false}，所以不会出现。 rowID 固定为 session-banner；estimatedHeight: 82 只是给虚拟滚动的初始高度。如果不再支持 side-chat session 作为普通 pane 打开，它可以视为历史/兼容 row。 直接渲染 thread-session-banner article，不经过 turn / trace item。
user-turn 主时间线里的用户输入气泡。它代表“用户正式发给 session 的一轮输入”，不是所有用户输入都会在顶层出现。 turn.kind === "user" 时产生；如果这个 user turn 是 streamInsertion，会被跳过，之后嵌到对应 assistant row 内部。 turn 是完整用户数据；turnIndex 用来判断 diff card 放在用户行还是后面的 assistant 行；estimatedHeight 会按文本长度、附件、diff 估算。 UserTurnArticle。如果它有独立 diff summary，还会旁挂 TurnDiffCard。
process-header assistant 过程区的“抽屉门”。它不是内容本身，而是把 reasoning、tools、workflow 等过程内容压缩成一条 Processed 摘要。 assistant turn 完成后，如果最终回复前面有足够的过程内容，并且不只是很短的一句 reasoning，就会在 assistant row 前面插入。 blocks 是抽屉里有哪些 section；expanded 决定是否露出 process-item；collapsing 用于完成态自动折叠动画；turnID 绑定原 assistant turn。 AssistantProcessTraceHeader。点击它会更新 processTraceExpansionByTurnID，从而让下一次 displayRows 决定是否插入 process-item。
process-item 抽屉展开后看到的真实过程内容。可以把它理解成“从 assistant turn 里提前搬出来的单条 trace item”。 只有对应 process-header 处于展开状态，或者正在折叠动画中，才会逐条插入；折叠完成后这些 row 会从 displayRows 里消失。 item 是原始 trace item；section 决定外层样式和 aria label；itemID 用于稳定定位；rowID 包含 turn、block、item，避免同一 item 出现在不同位置时冲突。 TraceItemView。它还会继续按 item.kind 分发到 tool、text、patch、image、reasoning 等具体 renderer。
assistant assistant turn 的主体行。最终回复一定要回到这里，因为复制、fork、side chat、branch 这些动作都围绕最终 assistant message。 只要 assistant turn 有可显示 item、临时提示、或插入的 user turn，就会产生。流式早期可能只有 ephemeralHint；完成后过程内容可能被移到 Processed 区，只留下最终回复。 renderedItems 是主体区真正显示的 trace items；processPrefixItems 是同一 assistant run 的前序过程；insertedUserTurns 是插队用户输入；ephemeralHint 是无内容时的 Thinking/Running 提示。 ephemeralHint 存在时渲染 AssistantTurnPlaceholder；否则走 AssistantTurnSectionsWithStreamInsertions，再进入 section / trace item 渲染。
permission-request 一个被包装成 thread row 的阻塞交互。它不是 turn，但用户需要在消息流里直接处理它，所以被放进 displayRows。 遍历完 turns 后，如果 pendingPermissionRequests[0] 存在，并且当前没有正在 resolving，就追加到队尾；用户允许/拒绝或进入 resolving 后消失。 rowID 用权限请求 id 生成；estimatedHeight: 420 给虚拟滚动一个较大的初始空间，因为权限卡片内容较高。 PermissionRequestInlinePrompt，使用当前 pending requests 和 resolving/error 状态渲染允许/拒绝操作。
读一条 row 时按这个顺序看
  • 先看 row.kind：它决定这行属于“消息主体、过程抽屉、抽屉内容、权限交互、上下文提示”哪一类。
  • 再看 rowID：它决定 React key、虚拟列表高度缓存、滚动恢复是不是能稳定追踪这一行。
  • 如果是 assistant 相关 row，再看 turnID / turn：它们把拆出来的过程行重新关联回同一个 assistant turn。
  • 最后看 estimatedHeight：它不决定最终视觉高度，只是虚拟滚动第一次排版的估算值。

内部顺序：user turn 怎么进 displayRows

普通用户消息最简单，估算高度后直接入队。

if (turn.kind === "user") {
  if (hasStreamInsertionTarget(activeTurns, turn)) return;

  rows.push({
    kind: "user-turn",
    rowID: turn.id,
    turn,
    turnIndex,
    estimatedHeight: estimateUserThreadRowHeight(turn)
  });

  return;
}

为什么 displayRows 需要 rowID 和 estimatedHeight

它们让 React identity、虚拟滚动和滚动恢复能稳定工作。

用 displayRows 读 ThreadView 的心智模型
  • activeTurns 是业务历史；displayRows 是屏幕历史。两者不是一一对应。
  • 一个 assistant turn 可能变成 process-header、多个 process-item、一个 assistant row。
  • stream insertion 的 user turn 会从普通位置跳过，然后嵌回对应 assistant row 内部。
  • 权限请求不属于 turns，但会被追加成 row，因此用户看到它像 thread 的一部分。
  • displayRows 之后的所有系统都更关心 row：虚拟滚动、滚动恢复、动画状态和最终 UI 分发。
