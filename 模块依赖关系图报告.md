# Fanfande Studio 模块依赖关系图报告

## 1. 报告目的
这份报告聚焦“模块如何依赖、数据如何流动、主入口如何串联”，用于补充上一份框架分析报告。

## 2. 总体依赖结构
从代码入口看，项目整体依赖关系可以概括为：

```text
root workspace
  ├─ packages/desktop
  │    ├─ main
  │    ├─ preload
  │    ├─ renderer
  │    └─ shared
  ├─ packages/fanfandeagent
  │    ├─ server
  │    ├─ session
  │    ├─ tool
  │    ├─ project
  │    ├─ permission
  │    ├─ provider
  │    ├─ pty
  │    ├─ mcp
  │    ├─ skill
  │    └─ database
  ├─ packages/site
  └─ packages/monitor
```

其中真正的核心依赖链是：

```text
desktop/renderer
  ↓ IPC / bridge
desktop/main
  ↓ HTTP / SSE / WebSocket
fanfandeagent/server
  ↓ usecases
fanfandeagent/session + project + tool + permission + provider
  ↓
database / bus / pty / mcp / skill
```

## 3. Desktop 端依赖关系

### 3.1 主启动链路
根据 `packages/desktop/src/main/index.ts`，桌面启动顺序是：

```text
index.ts
  ├─ ensureManagedAgentRunning()
  ├─ createApplicationMenus()
  ├─ registerIpcHandlers()
  └─ createWindow()
```

即：
- 先尝试拉起本地 Agent
- 再注册菜单
- 再注册 IPC
- 最后创建桌面窗口

这说明桌面端主进程是整个桌面壳的装配中心。

### 3.2 Main 内部依赖图
`packages/desktop/src/main/ipc.ts` 是主进程最大枢纽之一，依赖了：

```text
ipc.ts
  ├─ agent-client.ts
  ├─ appearance-config.ts
  ├─ external-editors.ts
  ├─ folder-workspaces.ts
  ├─ git.ts
  ├─ pty-proxy.ts
  ├─ workspace-diff.ts
  ├─ workspace-files.ts
  ├─ workspace-watch.ts
  ├─ window-state.ts
  ├─ ../shared/desktop-ipc-contract
  └─ ./types
```

可以把它理解成：
- `ipc.ts` = 桌面主进程能力统一出口
- Renderer 不直接碰 Node/Electron 能力
- 都通过 IPC 合法访问

### 3.3 Renderer 顶层依赖图
根据 `packages/desktop/src/renderer/src/App.tsx`：

```text
App.tsx
  ├─ useDesktopShell()
  ├─ useAgentWorkspace()
  ├─ useGlobalSkills()
  ├─ useSettingsPage()
  ├─ WorkbenchShell
  └─ TerminalAreaHost
```

所以 Renderer 顶层不是“大量业务直接写在 App”，而是：
- App 负责总装
- 核心业务通过 hooks 拆分
- 复杂界面通过 workbench / sidebar / terminal 等区域分治

### 3.4 Renderer 内部核心依赖
从 `use-agent-workspace.ts` 的导入关系看，它是前端业务编排中心：

```text
use-agent-workspace.ts
  ├─ workspace-store
  ├─ workspace-loading-controller
  ├─ session-stream-controller
  ├─ session-lifecycle-controller
  ├─ composer-controller
  ├─ review-panel-controller
  ├─ stream-permission-controller
  ├─ workbench-state
  └─ workbench-tab-controller
```

说明前端状态不是散的，而是围绕“workspace”统一组织。

这基本形成了一个前端内部依赖层次：

```text
App
  ↓
useAgentWorkspace
  ↓
controller hooks
  ↓
workspace store / session bridge / review state / workbench state
  ↓
components
```

### 3.5 Desktop 共享层
`packages/desktop/src/shared/desktop-ipc-contract.ts` 被：
- `main/ipc.ts`
- `preload/index.ts`
- `renderer/*`
共同依赖。

说明 `shared` 在 desktop 包中扮演“协议单一事实来源”角色：

```text
shared contract
  ├─ main 使用
  ├─ preload 使用
  └─ renderer 使用
```

这能减少 IPC 参数和事件名称漂移。

## 4. Agent 端依赖关系

### 4.1 Server 装配图
根据 `packages/fanfandeagent/src/server/server.ts`：

```text
server.ts
  ├─ routes/projects.ts
  ├─ routes/permissions.ts
  ├─ routes/pty.ts
  ├─ routes/debug.ts
  ├─ routes/settings.ts
  ├─ routes/session.ts
  ├─ pty/registry.ts
  └─ util/log.ts
```

Server 层只做：
- Hono app 创建
- 中间件装配
- 路由挂载
- websocket 升级
- 错误处理

这是很标准的“外壳层”。

### 4.2 Route -> Usecase 关系
例如 `packages/fanfandeagent/src/server/routes/session.ts`：

```text
routes/session.ts
  └─ usecases/session.ts
```

即路由层只负责：
- parse request
- validate body
- call usecase
- return response

不是把业务直接堆在 route 文件里。

### 4.3 Session 用例依赖图
从 `server/usecases/session.ts` 可归纳出：

```text
server/usecases/session.ts
  ├─ project/project.ts
  ├─ project/instance.ts
  ├─ session/session.ts
  ├─ session/message.ts
  ├─ session/prompt.ts
  ├─ session/running-state.ts
  ├─ session/diff.ts
  └─ server/usecases/session-stream.ts
```

这里可以看出后端会话功能分为两层：
- `usecases/session.ts`：HTTP 用例组织
- `session/*`：真正的领域行为

### 4.4 Session 核心处理链
从 `packages/fanfandeagent/src/session/prompt.ts` 与 `processor.ts` 的依赖关系可整理成：

```text
prompt.ts
  ├─ session/session.ts
  ├─ session/processor.ts
  ├─ session/system.ts
  ├─ session/diff.ts
  ├─ session/context-window.ts
  ├─ session/title.ts
  ├─ running-state.ts
  ├─ orchestrator.ts
  ├─ provider/provider.ts
  ├─ skill/skill.ts
  ├─ snapshot/snapshot.ts
  └─ project/instance.ts
```

```text
processor.ts
  ├─ session/llm.ts
  ├─ session/message.ts
  ├─ session/session.ts
  ├─ permission/permission.ts
  ├─ provider/provider.ts
  ├─ project/instance.ts
  └─ session/orchestrator.ts
```

组合起来，核心执行链大致是：

```text
HTTP session stream request
  ↓
server/usecases/session-stream.ts
  ↓
session/prompt.ts
  ↓
session/processor.ts
  ├─ llm.ts
  ├─ permission.ts
  ├─ tool execution
  └─ message persistence
  ↓
event-store.ts / live-stream-hub.ts
  ↓
SSE stream back to desktop
```

### 4.5 Tool Registry 依赖图
`packages/fanfandeagent/src/tool/registry.ts` 很清晰：

```text
tool/registry.ts
  ├─ builtin file tools
  ├─ builtin search tools
  ├─ builtin shell tools
  ├─ builtin LSP tools
  ├─ builtin skill tools
  ├─ builtin subagent tools
  ├─ mcp/manager.ts
  └─ project/instance.ts
```

这个 registry 的职责是：
- 聚合内置工具
- 注入 MCP tools
- 叠加 custom tools
- 做重名校验
- 按名称查找工具

所以工具体系不是直接散落执行，而是统一经 `registry.ts` 暴露。

### 4.6 Permission 依赖位置
`packages/fanfandeagent/src/permission/permission.ts` 依赖了：

```text
permission.ts
  ├─ database
  ├─ config
  ├─ tool/registry
  ├─ agent/agent
  ├─ session/message
  ├─ session/orchestrator
  ├─ session/session
  └─ project/instance
```

说明权限系统不是 UI 附属功能，而是深度嵌入在工具/会话/项目上下文中的中枢模块。

## 5. 横向基础设施依赖

### 5.1 Project / Instance
`project/instance.ts` 在大量模块中被引用：
- `provider`
- `tool`
- `permission`
- `skill`
- `session`
- `lsp`
- `mcp`

这意味着 `Instance` 是后端运行时上下文容器。

可以把它看成：

```text
Instance
  = 当前项目作用域
  = 当前运行态上下文
  = 多模块共享状态入口
```

### 5.2 Database
`database/Sqlite.ts` 被以下领域频繁依赖：
- `session`
- `project`
- `permission`
- `config`
- `memory-store`
- `event-store`

所以数据库层不是“单独 ORM 包”，而是后端多个核心域的共用持久化基座。

### 5.3 Bus / Event 系统
依赖关系显示：
- `session/event-store.ts` 负责运行事件落盘/投影
- `session/live-stream-hub.ts` 负责实时订阅
- `bus/*` 与 `pty/events.ts` 负责模块间广播

因此系统同时存在两类事件流：
- **会话运行事件流**：服务于 SSE/流式输出
- **模块广播事件流**：服务于内部解耦

## 7. 实际上的核心中枢模块
从依赖密度看，几个真正的中枢是：

### Desktop 侧
- `packages/desktop/src/main/ipc.ts`
- `packages/desktop/src/renderer/src/app/use-agent-workspace.ts`
- `packages/desktop/src/shared/desktop-ipc-contract.ts`

### Agent 侧
- `packages/fanfandeagent/src/server/server.ts`
- `packages/fanfandeagent/src/server/usecases/session.ts`
- `packages/fanfandeagent/src/session/prompt.ts`
- `packages/fanfandeagent/src/session/processor.ts`
- `packages/fanfandeagent/src/tool/registry.ts`
- `packages/fanfandeagent/src/project/instance.ts`

## 8. 依赖结构评价

### 8.1 优点
- 分层总体清晰
- 协议边界明确
- Route / Usecase / Domain 分离较好
- Tool 体系集中管理
- Desktop Renderer 没有直接耦合底层系统 API

### 8.2 复杂点
- `ipc.ts` 依赖面很广，属于 desktop 侧高耦合枢纽
- `session/prompt.ts` 与 `session/processor.ts` 承担了非常重的后端核心责任
- `project/instance.ts` 是强共享上下文点，后续若继续膨胀，容易成为隐式耦合源

## 10. 结论
如果说上一份报告回答的是“这个项目有哪些层”，那这份报告回答的是：

**“这个项目真正靠哪些中枢模块把所有层串起来。”**

结论是：
- Desktop 侧的核心中枢是 `ipc.ts + use-agent-workspace.ts + shared contract`
- Agent 侧的核心中枢是 `server/usecases/session.ts + session/prompt.ts + session/processor.ts + tool/registry.ts + project/instance.ts`
- 整个系统本质上是一个“桌面 UI 壳 + 本地 Agent Runtime + 流式事件总线”的组合架构
