

轻量裁剪？


**需求目标**

- 系统目标不是“生成一段摘要”，而是在上下文预算受限时，尽量保留持续工作的能力。
- 完整历史必须外部持久化；模型上下文只保留继续执行所需的最小工作集。
- 压缩应优先采用低损耗方式，只有接近阈值时才退化到语义摘要。

**分层模型**

- 完整真相层：持久化 transcript、会话日志、session memory 文件，作为可回放的外部事实源。
- 模型工作层：每轮实际发送给模型的消息窗口，默认仅取最近一个 compact_boundary 之后的消息。
- 热状态层：近期原文尾部、最近读取文件、plan、已发现工具、MCP 指令、session start hooks 结果等运行态信息。

**功能需求**

- 查询前必须按成本从低到高执行减载链路：snip -> microcompact -> context collapse（若开启）-> auto compact。
- microcompact 必须优先清理高体积、低长期价值的旧 tool_result，并尽量不破坏 prompt cache。
- full compact 必须生成结构化 summary，并保留最近原文尾部；尾部需有最小 token 和最少文本消息数下限。
- 每次 full compact 后必须插入 compact_boundary，后续默认只看该边界之后的消息。
- session memory 必须由后台 forked subagent 持续增量维护，不能等到爆上下文时才临时总结。
- auto compact 时必须优先尝试用 session memory 作为长期记忆来源；不可用时再回退到传统摘要。
- 压缩完成后必须回灌执行现场，包括近期文件、plan、plan mode、已调用技能、异步 agent 状态、工具列表、MCP 指令、session start hooks 上下文。
- summary 中必须显式提供 transcript 路径或等价入口，让系统在需要精确细节时可回读外部真相。

**约束与不变量**

- 不能拆坏 tool_use / tool_result 配对。
- 不能拆坏同一 assistant message.id 下的 thinking 与正文合并关系。
- 不能让压缩本身导致缓存命中率明显下降；有缓存时优先走 cache edit 类方案。
- 子代理如 session_memory、compact 不应递归触发自身压缩。
- compact 请求自身若超长，系统应支持剥离媒体、裁剪最旧消息组、有限次重试和失败熔断。

**验收标准**

- 压缩后，模型无需用户重复背景即可继续当前任务。
- 近期工作现场仍完整：最近文件、计划、工具/MCP 状态可直接继续使用。
- 长期历史虽不在 prompt 中，但可通过 transcript/session memory 找回。
- 大多数情况下应先通过轻量减载解决预算压力，只有必要时才做 full compact。
- 压缩后的消息序列仍满足 API 协议要求，不出现 orphaned tool_result、thinking 丢失或恢复失败。
