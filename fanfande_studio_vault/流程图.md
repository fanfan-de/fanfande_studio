## 流程图解读

| 层级 | 职责 | 关键函数 |
|------|------|----------|
| **入口层** | 分发三种用户输入类型 | `prompt()`, `shell()`, `command()` |
| **prompt() 流程** | 前置准备：状态检查、消息构建、系统提示词组装、工具集解析 | `assertNotBusy()`, `resolvePromptParts()`, `resolveTools()` |
| **loop() 核心循环** | 与 LLM 的多轮交互，包含工具调用、权限校验、压缩、步数限制 | `LLM.call()`, `Tool.execute()`, `SessionCompaction.run()` |
| **工具解析** | 从多个来源（内置、MCP、Plugin）合并可用工具 | `ToolRegistry`, `MCP`, `Plugin` |
| **子任务** | `TaskTool` 可以派生子 Session，形成递归 | `TaskTool` → 子 `prompt()` |
| **收尾** | 生成摘要、恢复状态、广播事件 | `SessionSummary.generate()`, `Bus.emit()` |

> **核心洞察**：这个文件本质上是一个 **AI 对话引擎**，`loop()` 是心脏。当前 `loop()` 函数约 350 行，承担了太多职责（LLM 调用、工具执行、压缩判断、步数控制、子任务分发），这是重构的首要目标——应按流程图中的各个判断分支拆成独立子函数。
