# `src/provider` 模块架构分析

## 整体架构层次

```
┌─────────────────────────────────────────────────────────────┐
│                    应用层 (Session/Agent)                    │
├─────────────────────────────────────────────────────────────┤
│                   提供者抽象层 (Provider)                     │
│  ┌──────────┐ ┌──────────┐ ┌──────────┐ ┌──────────┐      │
│  │  认证管理 │ │ 模型管理  │ │ 消息转换  │ │ 错误处理  │      │
│  │ (auth.ts) │ │(provider.ts)│(transform.ts)│ (error.ts)│      │
│  └──────────┘ └──────────┘ └──────────┘ └──────────┘      │
├─────────────────────────────────────────────────────────────┤
│                     适配层 (SDK适配器)                       │
│  ┌──────────────────────────────────────────────────────┐  │
```

## 核心组件详解

### 1. **提供者管理器 (provider.ts)**
**职责**：统一管理所有AI提供者的生命周期和状态
- **状态管理**：使用 `Instance.state()` 单例模式维护全局状态
- **SDK工厂**：动态加载和缓存AI SDK（OpenAI、Anthropic、Google等）
- **模型发现**：从多个源聚合模型数据
- **优先级系统**：智能选择默认模型和小型模型

**关键设计**：
- **懒加载**：SDK按需加载，使用哈希缓存
- **合并策略**：多源配置按优先级合并（环境变量 > 配置文件 > 插件 > 认证）
- **过滤机制**：支持黑名单/白名单，禁用实验性模型

### 2. **认证管理器 (auth.ts)**
**职责**：统一处理提供者认证流程
- **多认证方式**：支持OAuth和API Key两种模式
- **插件集成**：通过插件系统扩展自定义认证
- **状态跟踪**：管理OAuth流程的pending状态

**流程**：
```
授权请求 → OAuth回调 → 令牌存储 → 认证信息持久化
```

### 3. **模型数据源 (models.ts)**
**职责**：获取和缓存模型元数据
- **外部集成**：从 `models.dev` API获取最新模型信息
- **本地缓存**：支持离线使用和性能优化
- **自动刷新**：定时更新模型数据（默认每小时）
- **快照支持**：构建时生成静态快照

### 4. **消息转换器 (transform.ts)**
**职责**：处理不同提供者间的API差异
- **消息标准化**：统一消息格式，处理特殊字符和结构
- **参数适配**：针对不同提供者调整温度、top_p等参数
- **变体管理**：处理推理力度的不同表示方式
- **模式检测**：根据模型ID自动应用正确的转换规则

### 5. **错误处理器 (error.ts)**
**职责**：统一解析和处理提供者错误
- **错误分类**：区分上下文溢出、配额问题、API错误等
- **提供者适配**：针对不同提供者解析错误消息
- **重试策略**：判断错误是否可重试
- **用户友好**：将技术错误转换为用户可理解的消息

## 数据流

```
1. 初始化阶段
   models.dev API → 模型数据缓存 → Provider数据库
   │
   ├─ 配置文件 → 合并配置 → 提供者状态
   ├─ 环境变量 → 检测API Key → 激活提供者
   ├─ 认证系统 → 加载令牌 → 提供者认证
   └─ 插件系统 → 扩展提供者 → 自定义逻辑

2. 运行时阶段
   用户请求 → 选择模型 → 获取SDK → 消息转换
   │
   ├─ 认证检查 → OAuth流程 → 令牌刷新
   ├─ 参数调整 → 提供者特定配置 → API调用
   ├─ 错误处理 → 错误分类 → 重试/反馈
   └─ 结果缓存 → 性能优化 → 响应返回
```

## 设计模式应用

### 外观模式 (Facade)
- `Provider` 命名空间为外部模块提供统一接口
- 隐藏底层20+提供者的复杂性

### 策略模式 (Strategy)
- `CUSTOM_LOADERS`：每个提供者可定义自定义加载逻辑
- `ProviderTransform`：不同提供者使用不同的消息转换策略

### 适配器模式 (Adapter)
- `BUNDLED_PROVIDERS`：包装不同SDK到统一接口
- 消息格式适配：将标准消息转换为提供者特定格式

### 工厂模式 (Factory)
- SDK工厂：根据npm包名动态创建提供者实例
- 模型工厂：根据配置创建模型实例

### 状态模式 (State)
- `Instance.state()`：全局状态管理
- 认证状态机：管理OAuth流程的不同状态

### 观察者模式 (Observer)
- 插件系统：插件可监听提供者状态变化
- 配置变更：自动重新加载提供者配置

## 扩展机制

### 插件系统
```typescript
// 插件可扩展提供者认证和配置
plugin.auth = {
  provider: "custom-provider",
  methods: [...],
  loader: async () => ({...})
}
```

### 自定义加载器
```typescript
// 提供者特定的初始化逻辑
CUSTOM_LOADERS["custom-provider"] = async (provider) => {
  return { autoload: true, options: {...} }
}
```

### 配置继承
```typescript
// GitHub Copilot Enterprise 继承自 GitHub Copilot
database["github-copilot-enterprise"] = {
  ...githubCopilot,
  id: "github-copilot-enterprise",
  // 继承模型并修改providerID
}
```

## 关键特性

1. **多源配置合并**：环境变量、配置文件、认证系统、插件四层配置
2. **智能模型选择**：基于成本、能力和使用历史自动选择模型
3. **提供者特定优化**：20+提供者的深度优化（区域前缀、认证头等）
4. **错误恢复**：自动重试、降级策略、用户友好错误
5. **性能优化**：SDK缓存、模型数据缓存、懒加载
6. **扩展性**：插件系统支持自定义提供者和认证方式

## 依赖关系

```
src/provider/
├── → @ai-sdk/* (20+ SDK包)
├── → @opencode-ai/util/error
├── → src/config (配置管理)
├── → src/auth (认证系统)
├── → src/plugin (插件系统)
├── → src/util (工具函数)
├── → src/project (实例状态)
└── → src/flag (功能标志)
```

这个架构设计使 OpenCode 能够灵活支持各种AI提供者，同时保持代码的简洁性和可维护性。
