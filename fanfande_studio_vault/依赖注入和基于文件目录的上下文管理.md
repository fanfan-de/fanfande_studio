工作区隔离的上下文容器

这两个 TypeScript 脚本展现的是 **"依赖注入 + 基于目录的上下文状态管理"** 模式，具体来说是 **"工作区/项目隔离的状态容器"** 设计。这是专为多工作区场景设计的独特状态管理方法。

## 一、核心设计理念

### 1. **基于目录的上下文隔离**
```typescript
// 每个项目目录有自己的独立状态容器
const recordsByKey = new Map<string, Map<any, Entry>>()
// key = 项目目录路径
// value = 该目录下所有状态的集合
```

### 2. **依赖注入（DI）模式**
```typescript
// 通过 Context 管理当前执行上下文
const context = Context.create<Context>("instance")

// 在任何地方都可以获取当前上下文
Instance.directory  // 获取当前项目的目录
Instance.project    // 获取当前项目信息
```

### 3. **按需创建 + 单例缓存**
```typescript
// State.create 确保每个目录下相同的 init 函数只执行一次
const exists = entries.get(init)
if (exists) return exists.state as S  // 返回缓存的单例
```

## 二、这是什么样的状态管理？

### **"项目级依赖注入容器"（Project-scoped DI Container）**

类比理解：
- 像 **Spring Framework** 的 ApplicationContext，但有多个（每个目录一个）
- 像 **依赖注入框架**（如 Angular 的 Injector），但是按项目目录分区
- 像 **工作区沙箱**，每个项目有自己的独立运行环境

## 三、核心组件解析

### 1. **State.ts - 状态注册表**
```typescript
// 职责：管理所有项目的状态生命周期
class StateManager {
  // 两层嵌套Map结构
  private registry: Map<        // 第一层：按项目目录组织
    string,                    // 项目目录路径
    Map<                       // 第二层：按初始化函数组织
      Function,                // init() 函数本身作为key
      { state: any; dispose?: Function } // 状态实例
    >
  > = new Map()
  
  // 创建状态（工厂方法）
  create(root: () => string, init: () => S, dispose?: Function) {
    return () => {  // 返回一个获取状态的函数
      const key = root()  // 获取当前项目目录
      // 确保单例：相同的目录+相同的init函数 => 返回同一个状态
    }
  }
  
  // 清理项目状态
  async dispose(key: string) {
    // 调用所有状态的dispose方法
    // 清理缓存
  }
}
```

### 2. **Instance.ts - 上下文管理器**
```typescript
// 职责：管理当前执行上下文和项目实例
class InstanceManager {
  // 项目实例缓存（按目录）
  private instances: Map<string, Promise<Context>> = new Map()
  
  // 上下文栈管理
  private context = Context.create<Context>("instance")
  
  // 提供执行上下文
  async provide<R>(options: {
    directory: string;      // 项目目录
    init?: () => Promise<any>; // 初始化逻辑
    fn: () => R;           // 要在上下文中执行的函数
  }): Promise<R> {
    // 1. 获取或创建项目实例
    // 2. 设置当前上下文
    // 3. 执行用户函数
    // 4. 返回结果
  }
  
  // 便捷访问器
  get directory() { return context.use().directory }
  get project() { return context.use().project }
  
  // 创建项目状态（委托给State）
  state<S>(init: () => S, dispose?: Function) {
    return State.create(() => this.directory, init, dispose)
  }
}
```

## 四、工作流程示例

```typescript
// 1. 定义状态类型
interface ProjectConfig {
  apiKey: string;
  theme: string;
  models: string[];
}

// 2. 创建状态（惰性初始化）
const useConfig = Instance.state(() => ({
  apiKey: "default",
  theme: "dark",
  models: ["gpt-4", "claude-3"]
}));

// 3. 在项目上下文中使用
async function processProject(projectDir: string) {
  return await Instance.provide({
    directory: projectDir,
    async init() {
      // 项目初始化逻辑
      console.log(`初始化项目: ${projectDir}`);
    },
    async fn() {
      // 在这里可以安全地访问当前项目的状态
      const config = useConfig(); // 自动获取当前项目的config状态
      console.log(`当前项目: ${Instance.directory}`);
      console.log(`配置: ${config.theme}`);
      
      // 其他业务逻辑...
      return "处理完成";
    }
  });
}

// 4. 处理多个项目
await processProject("/path/to/project-a");
await processProject("/path/to/project-b");

// 5. 清理项目状态
await Instance.dispose("/path/to/project-a");
```

## 五、设计模式分析

### 1. **组合模式**
- `State` 负责状态管理
- `Instance` 负责上下文管理
- 两者协作完成完整的场景

### 2. **代理模式**
```typescript
// Instance 的方法实际上代理到当前上下文
get directory() {
  return context.use().directory  // 代理到当前上下文
}
```

### 3. **工厂模式**
```typescript
// State.create 是状态工厂
const useConfig = State.create(() => dir, () => ({ /* 初始状态 */ }))
```

### 4. **观察者模式**
```typescript
// GlobalBus 用于发布-订阅
GlobalBus.emit("event", {
  type: "server.instance.disposed",
  // ...
})
```

## 六、适用场景

### 非常适合：
1. **IDE/编辑器插件** - 每个打开的项目需要独立状态
2. **多租户SaaS应用** - 每个租户（项目）有独立配置
3. **构建工具** - 同时处理多个项目的构建
4. **测试运行器** - 并行测试多个项目
5. **微服务开发环境** - 同时运行多个服务

### 使用示例：
```typescript
// VS Code扩展开发
class Extension {
  // 每个工作区有自己的设置
  static getSettings = Instance.state(() => 
    vscode.workspace.getConfiguration()
  );
  
  // 每个工作区有自己的语言客户端
  static getLanguageClient = Instance.state(() => 
    new LanguageClient(...)
  );
  
  // 激活扩展时
  activate(context: vscode.ExtensionContext) {
    // 当用户切换工作区时，自动切换上下文
    vscode.workspace.onDidChangeWorkspaceFolders(() => {
      Instance.disposeAll(); // 清理旧状态
    });
  }
}
```

## 七、优点

1. **强隔离性** - 项目间状态完全隔离
2. **自动清理** - 项目关闭时自动释放资源
3. **上下文感知** - 代码自动知道自己在哪个项目中执行
4. **惰性初始化** - 状态按需创建
5. **类型安全** - TypeScript 全程类型支持
6. **资源管理** - 明确的 dispose 生命周期

## 八、与传统状态管理的对比

| 特性 | Redux/Zustand | 这种方案 |
|------|--------------|----------|
| 状态隔离 | 全局单一状态树 | 按项目目录隔离 |
| 上下文感知 | 需要手动传递 | 自动通过执行栈管理 |
| 生命周期管理 | 手动清理 | 自动 dispose |
| 适用场景 | 单页面应用 | 多工作区/多项目应用 |
| 状态共享 | 全局共享 | 项目内共享，项目间隔离 |

## 总结

这是一个 **专为多工作区场景设计的企业级状态管理方案**，结合了：
- **依赖注入**（管理执行上下文）
- **单例模式**（确保状态唯一性）
- **资源池模式**（管理状态生命周期）
- **发布订阅**（事件通信）

它特别适合需要同时处理多个独立项目环境的工具和应用程序，如 IDE、构建系统、多租户应用等。这种设计在开发工具领域很常见，但在 Web 前端中相对少见。
