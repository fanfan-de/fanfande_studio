没问题！TypeScript 的模块引用（Module Resolution）是新手最容易“鬼打墙”的地方，但只要掌握了核心逻辑，其实非常清晰。

要彻底搞懂这个问题，你需要理解以下 **5 个核心概念**。

---

### 1. 基础：相对路径引用 (Relative Imports)
这是最原始、最通用的方式，不依赖任何配置。

*   **`./`**：代表**当前**文件所在的文件夹。
*   **`../`**：代表**上一级**文件夹。

**例子：**
假设你的文件结构如下：
```text
src/
├── config/
│   └── global.ts
└── services/
    └── user/
        └── login.ts  <-- 你在这里写代码
```

在 `login.ts` 中引用 `global.ts`：
```typescript
// 1. 先跳出 user 目录 (../) 到 services
// 2. 再跳出 services 目录 (../) 到 src
// 3. 进入 config 目录
// 4. 找到 global
import { Global } from "../../config/global"; 
```
**缺点**：当目录层级很深时，你会写出这种代码：`../../../../utils/helper`，非常难看且难以维护（被称为 "Dot Hell"）。

---

### 2. 进阶：路径别名 (Path Alias) —— **推荐**
为了解决上面的“点点地狱”，TypeScript 允许你在 `tsconfig.json` 里定义“快捷方式”。这是现代前端项目的标配。

**第一步：配置 `tsconfig.json`**
```json
{
  "compilerOptions": {
    // 1. 设定根目录（通常是项目根目录）
    "baseUrl": ".", 
    // 2. 设定路径映射
    "paths": {
      // 这里的 "@/*" 是你自定义的名字，通常用 @ 表示 src 目录
      "@/*": ["src/*"] 
    }
  }
}
```

**第二步：使用别名**
无论你的文件在多深的目录里，引用方式永远统一：
```typescript
import { Global } from "@/config/global"; 
// TS 会自动把它翻译成： src/config/global
```

---

### 3. 导出与导入的两种姿势 (Named vs Default)
很多引用错误是因为**姿势不对**。

#### 姿势 A：命名导出 (Named Export) —— **推荐**
严谨，强制名字一致，利于重构。
```typescript
// A.ts
export const Global = { ... }; // 必须有 export 关键字

// B.ts
import { Global } from "./A"; // 必须加花括号 {}，名字必须叫 Global
```

#### 姿势 B：默认导出 (Default Export)
随意，但容易乱。
```typescript
// A.ts
const Global = { ... };
export default Global; // 一个文件只能有一个 default

// B.ts
import MyConfig from "./A"; // 不需要花括号，名字可以随便改（容易造成歧义）
```

> **你的代码中使用的是 `export const Global`，所以必须用 `import { Global }`。**

---

### 4. 索引文件 (Barrel File / index.ts)
这是一种让引用更整洁的高级技巧。你可以在文件夹里放一个 `index.ts` 作为“接待员”。

**目录结构：**
```text
src/
└── utils/
    ├── log.ts
    ├── date.ts
    └── index.ts  <-- 接待员
```

**`src/utils/index.ts` 内容：**
```typescript
export * from "./log";
export * from "./date";
```

**引用时：**
你不需要写 `import { Log } from "./utils/log"`，而是可以直接引用文件夹：
```typescript
import { Log, DateUtils } from "./utils"; // 自动寻找 index.ts
```

---

### 5. 鬼故事：`.d.ts` 类型声明
有时候你会遇到：`Cannot find module 'xyz' or its corresponding type declarations.`

这通常发生在引用第三方库（如 `lodash`, `express`）时。
*   **原因**：这个库是用 JS 写的，TS 看不懂。
*   **解决**：你需要下载它的“翻译文件”（类型声明文件）。
    ```bash
    bun add -d @types/lodash
    # 或者
    npm install -D @types/lodash
    ```
*   **如果是你自己的文件报错**（比如这次的 `global`）：那通常是因为 TS 找不到文件（路径错），或者文件里没有任何 `export`（TS 认为它不是一个模块）。

---

### 总结：你的 `Global` 报错自查清单

1.  **路径对不对？**
    *   VS Code 鼠标悬停在 `import` 路径上，按住 `Ctrl` (Mac是 `Cmd`) 点击。
    *   如果**能跳转**，说明路径是对的。
    *   如果**不能跳转**，说明路径错了（少了一层 `../` 或文件夹拼写错误）。

2.  **文件里导出没？**
    *   检查 `global.ts` 里是否有 `export const Global = ...`。

3.  **是否不仅是TS报错？**
    *   如果你用的是 `bun` 或 `ts-node`，它们通常能识别 `.ts` 文件。
    *   但如果你是编译成 JS 再运行，TS 只是负责检查。运行报错可能是编译配置没配好路径别名解析。

建议你现在就在项目根目录新建 `tsconfig.json` 并配置路径别名，这能极大提升开发体验！
