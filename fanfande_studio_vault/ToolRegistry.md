这段脚本是一个 **AI Agent 工具注册表 (Tool Registry)**。它的核心作用是**统一管理、加载和分发** AI 助手（Agent）可以调用的各种功能插件（如读写文件、执行命令、网页搜索等）。

以下是该脚本的具体功能拆解：

### 1. 核心职责
该脚本定义了一个名为 `ToolRegistry` 的命名空间，主要负责：
*   **静态工具集成**：加载内置的硬编码工具（如 `BashTool`, `EditTool`, `ReadTool` 等）。
*   **动态插件加载**：从本地目录或插件系统中扫描并加载自定义工具。
*   **工具过滤与分发**：根据当前使用的 **AI 模型类型**、**客户端类型**或**环境变量**，决定给 Agent 提供哪些工具。

---

### 2. 关键函数与逻辑

#### (1) 动态加载 (`state` 变量)
这部分代码通过 `Bun.Glob` 扫描项目中的 `tool/` 或 `tools/` 目录：
*   它会读取所有 `.js` 或 `.ts` 文件。
*   利用 `import(match)` 动态导入这些模块。
*   将这些外部定义的工具转换为系统统一的 `Tool.Info` 格式，存入 `custom` 数组。
*   同时也会加载通过 `Plugin.list()` 获取的插件工具。

#### (2) 格式转换 (`fromPlugin`)
这是一个适配器函数，将外部定义的 `ToolDefinition` 包装成系统内部使用的工具对象：
*   **参数校验**：使用 `zod` 对 Agent 传入的参数进行校验。
*   **结果截断**：通过 `Truncate.output` 处理工具执行后的返回结果。如果输出太长（超出 Token 限制），会自动截断并记录截断状态，防止 Agent 崩溃。

#### (3) 汇总所有工具 (`all` 函数)
这是工具的“大本营”，它合并了：
*   **基础工具**：`BashTool` (执行命令), `ReadTool` (读文件), `WriteTool` (写文件) 等。
*   **实验性工具**：根据 `Flag` 标志决定是否启用 `LspTool` (代码补全) 或 `BatchTool`。
*   **特定场景工具**：例如 `PlanEnterTool` 仅在 CLI 模式下启用。
*   **自定义工具**：之前动态加载的 `custom` 列表。

#### (4) 模型适配与过滤 (`tools` 函数)
这是最智能的部分，它根据 **AI 模型的身份** 来动态调整工具包：
*   **搜索限制**：`codesearch` 和 `websearch` 仅在特定 Provider（如 `opencode`）或开启特定 Flag 时可用。
*   **编辑模式切换**（非常关键）：
    *   如果模型是某些特定的 GPT 模型（通常是那些对 Patch 格式支持更好的模型），则只给它 `apply_patch` 工具。
    *   对于其他模型，则提供传统的 `edit` 和 `write` 工具。
    *   这种策略是为了**匹配不同模型的长处**（有的擅长写全文，有的擅长生成补丁）。

---

### 3. 工具列表概览
从代码中可以看到这个 Agent 拥有非常强大的能力：
*   **文件操作**：`ReadTool`, `WriteTool`, `EditTool`, `GlobTool` (文件搜索), `ApplyPatchTool` (应用补丁)。
*   **系统交互**：`BashTool` (终端执行), `TaskTool` (任务管理)。
*   **信息获取**：`WebFetchTool` (爬虫), `WebSearchTool` (搜索), `CodeSearchTool` (代码搜索), `GrepTool` (文本搜索)。
*   **辅助功能**：`QuestionTool` (提问), `TodoTool` (待办), `SkillTool` (技能调用), `LspTool` (编程语言服务)。

### 4. 总结
这个脚本就像是一个 **“武器库管理系统”**。当 Agent 准备开始工作时，它会告诉管理系统：“我是 GPT-4，我正在 CLI 环境下运行。” 管理系统就会根据这些信息，从武器库中挑选出最适合 GPT-4 且当前环境下可用的工具集（如 Bash、文件读写、网页搜索）打包发送给 Agent 使用。
