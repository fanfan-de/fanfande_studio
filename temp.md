C4 模型（Context, Containers, Components, Code）的核心思想是像“谷歌地图”一样对软件架构进行分层缩放。判断一个部分处于哪个层级，最关键的不是看它的名字，而是看它的**部署边界**、**运行环境**以及**受众群体**。

以下是快速判断和确定软件部分所属层级的指南：

### 1. 核心判断标准：层级定义的“边界”

| 层级                     | 关键词         | 判断问题                            | 常见例子                                  | 受众           |
| :--------------------- | :---------- | :------------------------------ | :------------------------------------ | :----------- |
| **L1: 上下文 (Context)**  | **生态圈**     | 这个系统在整个世界中扮演什么角色？它和哪些外部人和系统交互？  | 你的整套系统、用户、第三方支付、外部邮件服务                | 全员（技术 + 非技术） |
| **L2: 容器 (Container)** | **可独立部署单位** | 它是一个可以独立启动、停止或部署的进程/数据存储吗？      | Web 应用、微服务、数据库、移动 APP、定时脚本            | 架构师、开发人员、运维  |
| **L3: 组件 (Component)** | **代码逻辑分组**  | 在一个容器内部，代码是如何按逻辑功能划分的？（通常不独立运行） | Controller、Service、Repository、Auth 模块 | 架构师、开发人员     |
| **L4: 代码 (Code)**      | **实现细节**    | 这个功能的具体类图或数据结构是怎样的？             | Java 类、接口、数据库 Schema、算法实现             | 仅限开发人员       |

### 2. 如何具体确定某个部分的位置？

你可以通过以下三个维度的“漏斗”进行过滤判断：

#### 第一步：看“部署与运行”（区分 L2 和 L3）
这是最容易混淆的地方。
*   **判断准则：** 如果一个东西能**单独被打成一个包（如 .jar, .docker image, .exe）并在一个独立的进程/地址空间运行**，它就是 **L2 容器**。
*   **反之：** 如果它必须寄生在某个进程里（比如只是一个文件夹下的代码模块或 JAR 包里的一个类），它就是 **L3 组件**。
    *   *例子：* 一个 Spring Boot 微服务是 L2；而这个服务里的 `OrderService` 接口和实现类是 L3。

#### 第二步：看“视角范围”（区分 L1 和 L2）
*   **判断准则：** 它是你正在构建或负责的“主体系统”吗？
*   **L1 视角：** 把你的整个系统看作一个“黑盒”。你只关心谁在用它（用户）以及它求助于谁（第三方 API）。
*   **L2 视角：** 打开这个黑盒，看里面有哪些**大块头**（应用、数据库）在打配合。
    *   *注意：* 数据库（MySQL, Redis）在 C4 中明确属于 **L2 容器**，因为它是一个独立的存储进程。

#### 第三步：看“颗粒度”（区分 L3 和 L4）
*   **判断准则：** 我需要展示的是“逻辑块”还是“具体实现”？
*   **L3 视角：** 展示系统内部的“功能分区”，比如“用户认证模块”、“订单处理逻辑”。
*   **L4 视角：** 展示类之间的继承、聚合关系。
    *   *建议：* 绝大多数情况下，不要画 L4。代码本身就是最好的 L4。

### 3. 常见争议部分的归类建议

*   **微服务 (Microservice)：**
    *   如果你在画“公司所有业务的分布”，微服务可以被看作 **L1 系统**（因为太多了）。
    *   但在标准的单系统建模中，一个微服务就是一个 **L2 容器**。
*   **API 网关 (Gateway)：** 
    *   它是一个独立部署的组件，所以是 **L2 容器**。
*   **公共类库 (Shared Library)：**
    *   虽然它被打包，但它不独立运行，它会被合并进其他服务进程中。因此在 C4 中，它通常被视为 **L3 组件**（或在多个 L2 容器中出现的组件）。
*   **前端框架 (React/Vue)：**
    *   虽然代码在浏览器运行，但它是一个独立交付的逻辑单元，属于 **L2 容器**。

### 4. 实践中的“黄金法则”

1.  **先画 L1 和 L2：** 80% 的架构沟通只需要这两层。如果 L1/L2 讲不清楚，往下画也没用。
2.  **不要为了画而画：** 如果你的容器（比如一个简单的 Node.js API）内部非常扁平，就没必要画 L3 组件图。
3.  **不要混层：** 绝对不要在 L1 环境图中画出一个数据库，也不要在 L2 容器图中画出一个具体的 Java 类。保持每张图的“缩放比例”一致。

**总结：** 
- **L1** 关注**外部连接**；
- **L2** 关注**物理进程/独立应用**；
- **L3** 关注**逻辑模块**；
- **L4** 关注**代码实现**。
