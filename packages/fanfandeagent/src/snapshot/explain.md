这段代码定义了一个名为 `Snapshot` 的命名空间，其核心作用是**利用 Git 作为后端引擎，为特定的项目实例（Instance）提供文件快照、差异对比、备份与恢复功能**。

它并没有把项目变成一个普通的 Git 仓库，而是通过 Git 的底层命令（Plumbing commands）在后台静默地管理文件的版本状态。

以下是代码各个部分的详细功能分析：

### 1. 核心机制：Git 目录分离
*   **`gitdir()`**: 快照的 Git 数据并不存储在项目文件夹下的 `.git` 中，而是存储在系统全局数据目录（`Global.Path.data/snapshot/{project.id}`）下。
*   **工作区绑定**: 所有的 Git 命令都通过 `--git-dir` 指向上述私有目录，通过 `--work-tree` 指向项目的实际物理路径。这意味着它可以在不干扰用户原有版本控制系统的情况下，独立记录文件变化。

### 2. 主要功能方法

#### **A. 快照记录 (Tracking)**
*   **`track()`**:
    *   如果 Git 目录不存在，则执行 `git init` 初始化。
    *   执行 `git add .` 将当前所有文件加入暂存区。
    *   执行 `git write-tree`：**这是关键点**。它不创建提交（commit），而是直接生成一个代表当前目录树结构的哈希值（Hash）。这个哈希值就是“快照 ID”。

#### **B. 差异对比 (Diffing)**
*   **`patch(hash)`**: 找出当前工作区文件与指定快照哈希之间的差异，返回被修改过的文件路径列表。
*   **`diff(hash)`**: 生成标准的 Git diff 文本格式，展示具体的代码改动。
*   **`diffFull(from, to)`**: 深度对比。它遍历两个状态之间的每一个文件，获取修改前后的完整内容（`git show`）、新增行数和删除行数。这通常用于前端 UI 展示细粒度的文件对比。

#### **C. 恢复与撤销 (Restoring & Reverting)**
*   **`restore(snapshot)`**: 强制将整个工作区恢复到指定的快照状态（使用 `read-tree` 和 `checkout-index`）。
*   **`revert(patches)`**: 针对特定文件集合进行回滚。如果某个文件在快照中存在则恢复它，如果不存在（说明是后来新建的）则将其删除。

#### **D. 维护与清理 (Maintenance)**
*   **`cleanup()`**: 注册了一个定时任务，每小时运行一次。它执行 `git gc --prune=7.days`，清理超过 7 天的无用 Git 对象，防止快照数据占用过多磁盘空间。

### 3. 技术细节与工具类
*   **`unquote(path)`**: 这是一个工具函数，专门处理 Git 在输出路径时对特殊字符（如中文或特殊符号）进行的八进制转义。它将 Git 转义字符串还原为标准的 UTF-8 字符串。
*   **跨平台处理**: 设置了 `core.autocrlf=false`，确保在 Windows 和 Linux 之间处理换行符时不会破坏文件的一致性。
*   **Bun 集成**: 使用了 `import { $ } from "bun"`，利用 Bun 高效的 Shell 脚本执行能力来调用 Git 指令。
*   **类型安全**: 使用 `zod` 定义了 `Patch` 和 `FileDiff` 的模式，确保数据结构在序列化和传递过程中的准确性。

### 4. 总结：它被用来做什么？
这段代码通常出现在**在线编辑器（IDE）、低代码平台或自动化部署系统**中。它的用途包括：
1.  **自动保存记录**：在用户编写代码时定期生成 `write-tree`，实现类似 VS Code 的 "Timeline" 功能。
2.  **热更新/差异同步**：只计算发生变化的文件，从而实现增量同步。
3.  **误操作回滚**：当用户操作失误时，可以快速撤销到几分钟前的状态。
4.  **变更审计**：记录谁在什么时候改动了哪些代码行。

**代码亮点**：它巧妙地避开了 Git 的提交历史（Commit History）管理，只利用 Git 的内容寻址存储（Content-addressable storage）功能，这使得操作非常轻量且高效。
