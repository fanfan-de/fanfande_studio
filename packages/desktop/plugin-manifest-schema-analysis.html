插件 Manifest / Schema 实现说明

这份文档解释当前插件模块中 Manifest、MCP Server、Connector、Catalog Item 和 Installed Plugin 的 Zod schema 设计。 重点是理解这些 schema 如何作为插件系统的数据边界、运行时校验和归一化入口。

核心文件：packages/anyboxagent/src/plugin/plugin.ts 主要入口：PluginManifest line 402 测试：packages/anyboxagent/Test/plugin.test.ts

1. 这部分代码在解决什么问题

插件系统允许外部包声明 MCP server、技能、平台 connector 依赖、插件自带 connector 和 UI 展示信息。 这些声明来自本地插件包、内置插件包、registry metadata 或远程下载包，不能只靠 TypeScript 类型约束。 因此当前实现用 Zod 在运行时建立一套可验证的数据契约。

关键点

TypeScript 类型只保护编译期代码；Zod schema 保护运行时输入。插件 manifest 是外部 JSON， 必须经过 Zod 校验后才能进入 catalog、安装态和 MCP runtime 绑定流程。

plugin.json 插件作者提供原始 manifest。
PluginManifest Zod 校验顶层字段和能力声明。
PluginCatalogItem 归一化为 UI/API 可消费的插件市场条目。
InstalledPlugin 安装后写入 SQLite，记录生成的能力 ID。
Runtime Binding 同步到 global MCP config 和 connector 凭据系统。

2. 三层核心模型

Manifest

插件包里的原始声明，来自 .anybox-plugin/plugin.json 或兼容目录 .fanfande-plugin/plugin.json。

Schema: PluginManifest

Catalog Item

系统将 manifest、registry metadata、本地技能扫描结果归一化之后返回给插件市场 UI 和 API。

Schema: PluginCatalogItem

Installed Plugin

用户安装后的持久化记录，保存启用状态、生成的 MCP server、技能、connector 和插件配置。

Schema: InstalledPlugin

3. PluginManifest：插件作者写入 plugin.json 的契约

PluginManifest 是插件系统最重要的输入 schema。它定义 plugin.json 顶层允许出现哪些字段，哪些字段是必填，哪些能力声明会被系统识别。 

export const PluginManifest = z
  .object({
    name: z.string().min(1),
    version: z.string().min(1),
    description: z.string().min(1),
    author: PluginAuthor.optional(),
    homepage: z.string().optional(),
    repository: z.string().optional(),
    license: z.string().optional(),
    keywords: z.array(z.string()).optional(),
    interface: PluginInterface.optional(),
    mcpServers: z.array(PluginManifestMcpServer).optional(),
    skills: z.union([z.string(), z.array(z.string())]).optional(),
    connectorRequirements: z.array(Connector.ConnectorRequirement).optional(),
    connectors: z.array(PluginAppConnector).optional(),
    apps: z.array(PluginAppConnector).optional(),
    commands: z.union([z.string(), z.array(z.string())]).optional(),
    agents: z.union([z.string(), z.array(z.string())]).optional(),
  })
  .strict()
字段 含义 当前处理方式
name 插件名，也是生成插件 ID 的来源。 后续会 trim 并转小写，例如 Manifest Lab 会归一化为 manifest lab。
version 插件版本。 用于同一插件多版本选择、registry 匹配和安装记录。
description 插件基础描述。 作为 catalog 描述的 fallback。
interface 插件市场展示元数据。 允许额外字段透传，方便未来扩展 UI metadata。
mcpServers 插件自带 MCP server 声明。 安装后生成 plugin.<pluginID> 或 plugin.<pluginID>.<serverID>。
skills 插件技能根目录。 默认扫描 skills，只发现直接子目录里的 SKILL.md。
connectorRequirements 插件依赖平台已有 connector。 例如 Gmail/Feishu 插件依赖平台 connector，而不是声明插件私有 connector。
connectors 当前推荐的插件自带 connector 声明。 用于插件专属 API key、OAuth session、本地 wrapper 和运行时凭据注入。
apps 旧兼容字段。 会被归一化到 connectors 语义，公开路由仍使用 connectors。
commands, agents 保留字段。 schema 接收，但当前 v1 不执行。
.strict() 的意义

PluginManifest 顶层用了 .strict()，未知顶层字段会被拒绝。 这能保持插件包格式收敛，避免第三方插件依赖未定义字段。

4. PluginInterface：插件市场展示信息

PluginInterface 专门描述 UI 展示层 metadata。它和 manifest 顶层不同，使用 .passthrough()，也就是 interface 内部可以带额外字段。 

const PluginInterface = z
  .object({
    displayName: z.string().optional(),
    shortDescription: z.string().optional(),
    longDescription: z.string().optional(),
    developerName: z.string().optional(),
    category: z.string().optional(),
    capabilities: z.array(z.string()).optional(),
    websiteURL: z.string().optional(),
    privacyPolicyURL: z.string().optional(),
    termsOfServiceURL: z.string().optional(),
    defaultPrompt: z.union([z.string(), z.array(z.string())]).optional(),
    composerIcon: z.string().optional(),
    logo: z.string().optional(),
    iconUrl: z.string().optional(),
    thumbnailUrl: z.string().optional(),
    heroImageUrl: z.string().optional(),
    screenshots: z.array(z.string()).optional(),
    brandColor: z.string().optional(),
  })
  .passthrough()

后续 normalizeCatalogItem 会用这些字段生成插件市场卡片，例如： displayName 变成 catalog 的 name， shortDescription 变成 description， developerName 变成 publisher。

5. MCP Server Schema：插件自带 MCP 能力

MCP server 声明的 schema 是 PluginManifestMcpServer。它描述插件如何暴露一个直接可运行或可连接的 MCP server。 

const PluginManifestMcpServer = z
  .object({
    id: z.string().min(1).optional(),
    name: z.string().min(1).optional(),
    description: z.string().optional(),
    risk: PluginRisk.optional(),
    permissions: z.array(z.string()).optional(),
    tools: z.array(PluginToolPreview).optional(),
    configFields: z.array(PluginConfigField).optional(),
    runtime: PluginRuntimeTemplate,
    installReview: z.array(z.string()).optional(),
  })
  .strict()

最关键字段是 runtime，它指向 PluginRuntimeTemplate， 支持 stdio 和 remote 两类运行时。

stdio runtime

export const PluginStdioRuntime = z
  .object({
    transport: z.literal("stdio"),
    command: z.string().min(1),
    args: z.array(z.string()).optional(),
    env: z.record(z.string(), z.string()).optional(),
    cwd: z.string().min(1).optional(),
    timeoutMs: z.number().int().positive().optional(),
    toolPolicies: Config.McpToolPolicies,
  })
  .strict()

典型用途是启动插件包内的本地 MCP wrapper，例如 node ${PLUGIN_ROOT}/scripts/server.js。 安装时会把 placeholder 替换为插件配置和 PLUGIN_ROOT。

remote runtime

export const PluginRemoteRuntime = z
  .object({
    transport: z.literal("remote"),
    provider: Config.McpRemoteProvider.optional(),
    serverUrl: z.string().min(1).optional(),
    connectorId: z.string().min(1).optional(),
    authorization: z.string().min(1).optional(),
    headers: z.record(z.string(), z.string()).optional(),
    serverDescription: z.string().min(1).optional(),
    allowedTools: Config.McpAllowedTools,
    requireApproval: Config.McpRequireApproval,
    timeoutMs: z.number().int().positive().optional(),
    toolPolicies: Config.McpToolPolicies,
  })
  .strict()
直接 mcpServers 的 secret 边界

如果直接在 mcpServers.runtime.env 或 headers 中使用 secret placeholder， 安装同步后可能进入 MCP config。需要 API key 或 OAuth 的插件更适合使用 connectors， 由 connector 在运行时注入凭据。

6. Connector Schema：插件自带 connector

插件自带 connector 当前仍复用 PluginAppConnector 这个类型名，但公开语义已经从旧 app connector 迁移为 plugin connector。 它适合插件专属认证、插件包内本地 wrapper、远程 MCP 凭据注入和单独的连接状态管理。 

export const PluginAppConnector = z
  .object({
    appID: z.string().min(1).optional(),
    id: z.string().min(1).optional(),
    connectorID: z.string().min(1).optional(),
    name: z.string().min(1),
    description: z.string().optional(),
    icon: z.string().optional(),
    risk: PluginRisk.optional(),
    permissions: z.array(z.string()).optional(),
    tools: z.array(PluginToolPreview).optional(),
    configFields: z.array(PluginConfigField).optional(),
    credential: PluginAppCredential,
    runtime: PluginConnectorRuntimeTemplate,
    installReview: z.array(z.string()).optional(),
  })
  .strict()
  .transform((connector, ctx) => {
    const appID = connector.id ?? connector.connectorID ?? connector.appID
    if (!appID) {
      ctx.addIssue({
        code: "custom",
        message: "Plugin connector requires 'id' or legacy 'appID'.",
        path: ["id"],
      })
      return z.NEVER
    }

    return {
      ...connector,
      id: appID,
      appID,
    }
  })

这个 .transform() 是兼容设计的核心：新插件可以写 id， 迁移期插件可以写 connectorID，旧插件可以写 appID。 最终都会归一化为统一的 appID 字段，后续生成 ID 时不再区分来源。

插件写法 归一化结果 生成 ID
{ "id": "docs" } appID = "docs" plugin-connector:<pluginID>:docs
{ "connectorID": "docs" } appID = "docs" plugin-connector:<pluginID>:docs
{ "appID": "docs" } appID = "docs" plugin-connector:<pluginID>:docs

7. Credential Schema：API key 与 OAuth

插件 connector 的认证信息由 PluginAppCredential 表示，它是 API key 和 OAuth 两类 credential 的 union。 

export const PluginAppCredential = z.union([
  PluginOAuthAppCredential,
  PluginApiKeyAppCredential,
])

API key credential

API key credential 基于通用 config field 扩展，最终会被转换为 kind: "api_key"。 保存 API key 时，secret 写入 provider credential store，不写入 MCP config。 

export const PluginApiKeyAppCredential = PluginConfigField.extend({
  kind: z.literal("api_key").optional(),
}).transform((credential) => ({
  ...credential,
  kind: "api_key" as const,
}))

OAuth credential

OAuth credential 支持固定 client、动态客户端注册、scope 校验和 token placement。 它会通过 ProviderAuth 走 PKCE/OAuth 流程。 

const PluginOAuthAppCredentialBase = z
  .object({
    kind: z.literal("oauth"),
    label: z.string().min(1).default("OAuth"),
    clientID: z.string().min(1).optional(),
    clientSecret: z.string().min(1).optional(),
    authorizationURL: z.string().min(1),
    tokenURL: z.string().min(1),
    scopes: z.array(z.string().min(1)).min(1),
    revocationURL: z.string().min(1).optional(),
    tokenPlacement: PluginOAuthTokenPlacement.default({
      type: "authorization_bearer",
    }),
    authorizationParams: z.record(z.string(), z.string()).optional(),
    tokenParams: z.record(z.string(), z.string()).optional(),
    tokenEndpointAuthMethod: z
      .enum(["none", "client_secret_post", "client_secret_basic"])
      .optional(),
    registration: PluginOAuthClientRegistration.optional(),
    description: z.string().optional(),
  })
  .strict()
OAuth 校验点

OAuth credential 必须提供 clientID 或 registration。也就是说， 插件要么声明固定 OAuth client，要么声明动态客户端注册入口，否则 schema 会拒绝。

8. PluginCatalogItem：系统对外展示的标准形态

PluginCatalogItem 是 manifest 经过归一化后的公开模型。它不是插件作者直接写的格式， 而是插件市场、设置页、安装流程和项目选择使用的标准结构。 

export const PluginCatalogItem = z
  .object({
    id: z.string().min(1),
    name: z.string().min(1),
    description: z.string().min(1),
    longDescription: z.string().optional(),
    version: z.string().min(1),
    publisher: z.string().min(1),
    category: PluginCategory,
    icon: z.string().optional(),
    iconUrl: z.string().optional(),
    thumbnailUrl: z.string().optional(),
    heroImageUrl: z.string().optional(),
    screenshots: z.array(z.string()),
    tags: z.array(z.string()),
    brandColor: z.string().optional(),
    homepage: z.string().optional(),
    documentationUrl: z.string().optional(),
    risk: PluginRisk,
    permissions: z.array(z.string()),
    tools: z.array(PluginToolPreview),
    configFields: z.array(PluginConfigField),
    runtime: PluginRuntimeTemplate.optional(),
    mcpServers: z.array(PluginMcpServerCatalogEntry),
    skills: z.array(PluginSkillPreview),
    connectorRequirements: z.array(Connector.ConnectorRequirement),
    connectors: z.array(PluginAppConnector),
    apps: z.array(PluginAppConnector),
    installReview: z.array(z.string()).optional(),
    source: z.enum(["package", "registry"]).optional(),
    download: PluginPackageDownload.optional(),
    installable: z.boolean().optional(),
  })
  .strict()

归一化逻辑在 normalizeCatalogItem 中完成，主要做这些事情：

  • 把 manifest 的 name 归一化成 catalog id。
  • 把 mcpServers 补默认值，例如 server id 默认是 default，risk 默认是 medium。
  • 把新 connectors 和旧 apps 合并成同一套插件自带 connector 模型。
  • 如果是本地包，会扫描插件技能目录下的 SKILL.md 生成 skill preview。
  • 聚合 risk、permissions、tools 和 configFields，给 UI 一套可直接展示的数据。
  • 判断是否 installable：本地包可安装；registry 条目需要有 zip URL 和 sha256。

9. InstalledPlugin：安装态和数据库记录

InstalledPlugin 是用户安装插件之后写入 SQLite 表 installed_plugins 的记录。 它不再描述插件“可以提供什么”，而是描述插件“已经生成了什么”。 

export const InstalledPlugin = z
  .object({
    pluginID: z.string().min(1),
    version: z.string().min(1),
    enabled: z.boolean(),
    mcpServerID: z.string().min(1).optional(),
    mcpServerIDs: z.array(z.string()).optional(),
    skillIDs: z.array(z.string()).optional(),
    connectorIDs: z.array(z.string()).optional(),
    connectorRequirementIDs: z.array(z.string()).optional(),
    config: z.record(z.string(), z.string()),
    installedAt: z.number().int().positive(),
    updatedAt: z.number().int().positive(),
    lastDiagnostic: PluginDiagnostic.optional(),
    lastConnectorDiagnostics: z.record(z.string(), PluginDiagnostic).optional(),
    missingPackage: z.boolean().optional(),
  })
  .strict()
字段 作用
mcpServerIDs 安装后生成的插件 MCP server ID，包括普通 MCP server 和插件自带 connector 对应的 MCP binding。
skillIDs 插件带来的技能 ID，例如 plugin:manifest-lab:review。
connectorIDs 插件自带 connector 的 credential/provider ID，例如 plugin-connector:manifest-lab:docs。
connectorRequirementIDs 插件依赖的平台 connector ID，例如 connector:gmail:default。
config 安装时保存的插件配置，不是插件自带 connector 的 API key/OAuth session。
missingPackage 数据库仍有安装记录，但本地插件包缺失。UI 会显示 Package missing，并允许重新下载。

10. 输入 Schema：安装和更新

安装和更新接口刻意保持很小，只允许传入 enabled 和 config。 具体会生成哪些 MCP server、技能、connector，不由请求方决定，而是由 catalog item 和 manifest 派生。 

export const InstallPluginInput = z
  .object({
    enabled: z.boolean().optional(),
    config: z.record(z.string(), z.string()).optional(),
  })
  .strict()

export const UpdateInstalledPluginInput = z
  .object({
    enabled: z.boolean().optional(),
    config: z.record(z.string(), z.string()).optional(),
  })
  .strict()
为什么不让前端传 mcpServerIDs

生成 ID 是后端插件模块的职责。这样可以保证安装、更新、删除时都能用同一套规则计算和清理运行时绑定， 避免 UI 端构造出不一致的状态。

11. Manifest 读取和归一化入口

插件包读取入口是 safeReadPluginManifest(packageRoot)。它负责寻找 manifest 文件、执行 Zod 校验、 合并兼容字段，并把旧 apps 归一化为当前 connector 模型。 

function safeReadPluginManifest(packageRoot: string) {
  const manifestPath = [PLUGIN_MANIFEST_PATH, LEGACY_PLUGIN_MANIFEST_PATH]
    .map((candidate) => join(packageRoot, candidate))
    .find((candidate) => existsSync(candidate))
  if (!manifestPath) return undefined

  try {
    const raw = readFileSync(manifestPath, "utf8")
    const manifest = PluginManifest.parse(JSON.parse(raw))
    const manifestConnectors = normalizePluginConnectors(manifest)
    const compatApps = safeReadPluginAppCompat(packageRoot)
    ...
  } catch {
    return undefined
  }
}

这意味着不符合 schema 的插件不会进入 catalog，也不会被安装。当前实现选择了“读取失败返回 undefined”， 对用户表现为插件没有出现；这对稳定性有利，但对插件开发者调试不够直接。

12. 这套 Schema 的边界和设计取向

  • 强约束：Manifest 顶层、MCP server、runtime、catalog item、installed record 都使用 .strict()。
  • 兼容迁移：connectors 是当前语义，apps 和 plugin-app: ID 继续兼容。
  • 后端派生：安装态中的 server ID、skill ID、connector ID 都由后端根据 catalog 派生，不由前端直接指定。
  • 凭据隔离：插件 connector 的 API key/OAuth session 存在 Auth/ProviderAuth 中，运行时解析时才注入。
  • 项目隔离：项目只选择插件 ID；真正可用 MCP server 和 plugin skill 会根据已启用插件动态解析。
