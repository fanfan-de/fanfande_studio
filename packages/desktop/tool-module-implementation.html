基于当前工作区读取结果撰写，时间：2026-05-25。这里的“工具模块”主要指 packages/anyboxagent/src/tool 的 agent 侧工具定义、注册、过滤、权限与执行体系；同时覆盖 desktop 包中“Tools”管理界面的实现链路。

1. 范围与结论

问题结论
真正的工具实现在哪里？ packages/anyboxagent/src/tool。这里定义工具协议、内置工具、MCP/自定义工具合并、执行包装、权限接入、文件访问辅助。
desktop 的 Tools 页面做什么？ 只做内置工具全局可用性的读取、展示、切换、保存、重置；它不直接执行工具。
工具何时传给模型？ 每轮会话在 resolveTools 中读取注册表并按全局选择、agent allowlist、只读会话过滤，再包装成 AI SDK ToolSet。
权限如何决策？ 执行前由 createToolExecution 调 Permission.evaluate。决策来源包括工具声明、路径/风险推导、会话状态、审批记录、全局权限模式。
当前工作树的显著变化？ replace-text 已移除 path/search/replace/all 旧别名，当前参数为 file_path/old_string/new_string/replace_all；测试名称也从 alias 兼容转向 CRLF 保真。

2. 端到端调用链

用户会话启动/继续
  -> prompt.ts 调 resolveTools({ agent, sessionID, messageID, abort })
    -> ToolRegistry.tools()
      -> builtinTools()
      -> Mcp.tools()
      -> registry.state().custom
      -> assertUniqueToolNames()
    -> Config.getToolSelection("__global__")
    -> getToolAccessFailure()
      -> 全局内置工具禁用过滤
      -> agent.tools allowlist/denylist 过滤
      -> side-chat / read-only agent 过滤
    -> createToolExecution()
      -> item.init({ agent })
      -> Permission.evaluate()
      -> runtime.execute()
      -> maybePersistToolResult()
      -> runtime.toModelOutput() / 默认 text
    -> ai.tool({ title, description, inputSchema, needsApproval, execute, toModelOutput })
  -> AI SDK 流式产生 tool-call / tool-result
    -> processor.ts 维护 Message.ToolPart 状态并发 runtime events
    -> 如需审批：Permission.registerApprovalRequest()
    -> 用户批准/拒绝：Permission.resolveRequest()
      -> 批准时重新执行等待中的工具调用
      -> 拒绝时写入 denied 状态

3. 关键文件职责

文件角色主要内容关键点
anyboxagent/src/tool/tool.ts 工具协议与定义器 ToolInfo、ToolRuntime、ToolCapabilities、ToolOutput、define() 统一参数校验、validate/authorize、输出标准化、assessPermission 参数校验。
anyboxagent/src/tool/registry.ts 工具注册表 内置工具清单、平台 shell 工具、MCP 工具、自定义工具合并、唯一名检查 get() 同时匹配 id 和 aliases。
anyboxagent/src/tool/execution.ts 执行包装层 可用性过滤、只读会话判定、权限决策缓存、运行时上下文、输出持久化 模型不会直接调用 runtime，而是调用这里创建的 ToolExecution。
anyboxagent/src/session/core/resolve-tools.ts 模型工具集解析 把 registry 工具包装为 AI SDK ToolSet 只注册 provider-safe 名称：/^[a-zA-Z0-9_-]+$/。
anyboxagent/src/permission/permission.ts 权限与审批 风险推导、allow/ask/deny 决策、审批请求表、审批后工具补执行 side-chat 和 planning mode 对副作用工具有额外限制。
anyboxagent/src/tool/shared.ts 文件工具公共层 项目边界解析、真实路径检查、文本/二进制判定、原子写入、行范围格式化 写工具不能越过 active project boundary。
anyboxagent/src/server/usecases/settings.ts 设置用例 列出内置工具、保存工具选择、读写权限模式 将 Zod schema 转 JSON Schema 给 UI 展示。
desktop/src/renderer/src/app/tools/BuiltinToolsPage.tsx Tools 页面 分类导航、工具卡片、schema 展开、开关、保存/重置 纯展示/配置；执行仍在 agent。

4. 核心类型模型

类型字段/语义设计含义
ToolKind readwritesearchexecworkflowinteractiondelegationother 用于 UI 分组、权限风险推导、read-only session 过滤、parallel tool eligibility。
ToolCapabilities kind?, readOnly?, destructive?, concurrency?, needsShell? 工具声明自己的行为特征；权限层仍会结合输入路径和 intent 再判断。
ToolInfo id, title?, aliases?, capabilities?, maxResultSizeChars?, init() 注册表持有的静态工具定义；init() 生成实际 runtime。
ToolRuntime description, parameters, execute, validate?, authorize?, assessPermission?, describeApproval?, toModelOutput? 执行期能力。参数由 Zod schema 约束，description/inputSchema 暴露给模型。
Tool.Context sessionID, messageID, cwd?, worktree?, abort?, toolCallID? 把会话、项目目录、中断信号和工具调用 ID 注入工具。
ToolOutput text, title?, metadata?, data?, attachments? text 是默认模型可见输出；data/metadata 支持 UI、持久化和结构化消费。
ToolPermissionIntent action?, risk?, reason?, resource?, allowInPlanning?, forceAsk? 工具可在执行前声明本次调用的风险和建议动作，但最终由权限层裁决。

5. Tool.define 的封装语义

Tool.define(id, init, options) 返回 ToolInfo，但会在 init() 内替换 runtime 的 execute 和可选 assessPermission，使每个工具都获得一致的运行前处理。

阶段逻辑失败表现
参数解析 runtime.parameters.safeParse(args) 抛出包含工具 id 和 Zod 错误的异常；若定义了 formatValidationError 则用自定义文案。
validate 工具自定义业务校验，返回 string 或 { message } 表示失败。 抛出校验失败消息。
authorize 工具级硬授权检查，不同于全局权限审批。 抛出授权失败消息。
execute 调用原始 runtime.execute，并把 string 输出标准化为 { text }。 工具异常向上传递，由 processor 记为 failed/error。
assessPermission 如果存在，也先经过同一套参数解析。 避免权限评估看到未解析或 schema 外的输入。

6. 注册表与工具来源

来源实现说明
内置工具 ToolRegistry.builtinTools() 硬编码导入当前平台适用的工具；Windows 下加入 Git Bash、PowerShell、cmd、WSL Bash；macOS 下加入 macOS shell；其他平台当前无 shell 内置项。
MCP 工具 Mcp.tools() 由 MCP manager 转成 registry shape，工具 id 通常是 mcp__server__tool 形式。
自定义工具 registry.state().custom 通过 Instance.state 保存运行时可插入的工具定义，测试中也用它注入 custom tool。
合并结果 tools() [...builtinTools, ...mcpTools, ...custom]，合并后检查 id/alias 全局唯一。

当前内置工具族

族代表工具说明
交互/工作流ask-user-question, task tools处理用户问题、任务进度和任务状态。注意：enter-plan-mode.ts / exit-plan-mode.ts 文件存在，但当前未在 registry.ts 的 builtinTools() 中注册。
读/搜索read-file, glob, grep, list-directory, skill/resource tools多数应标记 readOnly=true 和 concurrency=safe。
写入replace-text, apply_patch项目内文本修改，通常 destructive=true、concurrency=exclusive。
Shell/Terminalgit_bash_command, powershell_command, cmd_command, wsl_bash_command, terminal tools需要 shell 或终端进程，风险和权限提示更高。
MCP 资源list_mcp_resources, list_mcp_resource_templates, read_mcp_resource作为内置只读工具出现，可在 read-only agent / side chat 中使用。
并行multi_tool_use_parallel只允许批量执行 read/search 且 readOnly=true、concurrency=safe 的子工具。
媒体/浏览generate-image, view-image, web-fetch图片、网页读取或生成能力。
委派spawn-subagent, read-subagent, wait-subagent, cancel-subagent子代理生命周期管理。

7. 会话解析与模型暴露

resolveTools 是模型可见工具集合的最后一道门。它不直接修改 registry，而是为本轮会话构造一个临时 ToolSet。

过滤条件函数/配置行为
全局内置工具选择 Config.getToolSelection("__global__") + isToolGloballyEnabled 只作用于内置工具。任一 id/alias 显式为 false 即禁用；空选择代表默认全部启用。
Agent 工具策略 agent.tools + isToolAllowedForAgent 如果有显式 false 则拒绝；如果出现任意 true，则变成 allowlist；空对象会拒绝所有工具。
只读会话 readOnlyToolsOnlyForSession side-chat 或 agent.toolPolicy === "read-only" 时，只保留 capabilities.readOnly === true 的工具。
Provider-safe 名称 /^[a-zA-Z0-9_-]+$/ 模型只获得合法函数名；如 multi_tool_use.parallel 旧别名不暴露给 provider，但 registry 仍可匹配。
重复名称 assertUniqueToolNames 与 resolveTools 中的 duplicate 检查 registry 层检查 id/alias 全局唯一；ToolSet 注册时再次防止 provider-safe 名称冲突。

8. 执行、权限、输出持久化

步骤实现说明
初始化 runtime item.init({ agent }) 允许工具根据 agent 信息改变行为。
构造上下文 runtimeContext(toolCallID) 注入 Instance.directory、Instance.worktree、AbortSignal、session/message/toolCall id。
权限评估 evaluatePermission() 先调用工具的 assessPermission，再交给 Permission.evaluate；非 ask 的决策按 toolCallID 缓存。
执行拦截 execute(args) deny 抛决策原因；ask 抛“requires approval”；allow 才调用 runtime。
大输出处理 ToolResultPersistence.maybePersistToolResult 超过工具的 maxResultSizeChars 时把大输出落盘，返回替换文本和 metadata，减少上下文压力。
模型输出转换 toModelOutput(output) 若输出已持久化，模型只看到替换文本；否则使用 runtime 自定义 toModelOutput 或默认 text。

9. 权限策略与审批流

权限决策顺序

  1. 准备权限表和审计表。
  2. 从输入与 intent 中抽取路径、命令、workdir、body，并推导风险。
  3. side-chat 中任何非 read-only 工具直接 deny。
  4. ANYBOX_PERMISSION=off/false/0/disabled 时直接 allow。
  5. 非 read-only 工具引用 active project boundary 外路径时 deny，风险为 critical。
  6. 如果当前 toolCallID 已有 approved/denied 审批记录，按记录复用。
  7. planning mode 下，副作用工具默认 deny，除非 intent 设置 allowInPlanning。
  8. 工具 intent 为 deny 时 deny；风险为 critical 时 deny。
  9. 工具 intent 为 ask 时，permission_mode=full_access 且未 forceAsk 则 allow，否则 ask。
  10. 工具 intent 为 allow 时 allow。
  11. fallback 使用 DEFAULT_ACTIONS；当前所有 kind 默认都是 allow。

审批对象与状态流

对象/事件来源作用
ToolApprovalDescriptorruntime.describeApproval 或 fallback生成用户看到的标题、摘要、路径/命令/正文详情。
permission_requestsregisterApprovalRequest持久化 pending/approved/denied 审批记录。
PermissionPartcreatePermissionPart写入对话消息 parts，连接 approvalID 与 toolCallID。
tool.call.waiting_approvalprocessor 或 permission registration通知 runtime/UI 工具正在等待用户。
permission.resolvedresolveRequest用户审批结果进入事件流。
tool.call.approved/completed/denied/failedcompleteApprovedRequest/denyApprovedRequest审批后补执行工具或写入拒绝状态。

10. 文件系统安全边界

能力实现安全语义
项目路径解析 resolveToolPath(inputPath) 相对路径基于 Instance.directory；绝对路径必须位于 active project boundary；拒绝 UNC 路径。
符号链接/真实路径防逃逸 resolveDeepestExistingPath + canonical roots 检查最终解析路径仍在 Instance.directory 或 Instance.worktree 内。
文本文件判定 扩展名黑名单 + 采样检测 NUL/控制字符比例 防止 read/write 工具误处理二进制文件。
设备路径阻断 BLOCKED_DEVICE_PATHS 与 /proc/*/fd 避免读取或写入特殊设备路径。
写入方式 先写临时文件，再 rename；异常时清理临时文件 尽量避免成功写入后出现半文件。
可读绝对路径 resolveReadableTextFilePath 允许只读工具引用项目外绝对路径；权限层也允许 read-only 工具外部路径。

11. replace-text 当前实现

replace-text 是当前工作树里被修改的工具文件。它定位于“单文件、精确字符串替换或创建一个新文本文件”，并明确建议多文件/结构化变更使用 apply_patch。

方面当前行为
参数 schema file_path: string、old_string: string、new_string: string、replace_all?: boolean。旧别名 path/search/replace/all 已从当前实现移除。
validate 如果 old_string === new_string，返回“No changes to make”。
创建文件 目标不存在且 old_string === "" 时创建文件；目标不存在但 old_string 非空时报错。
空文件填充 目标存在且 old_string === "" 时，只允许原文件 trim 后为空；否则报错，避免覆盖非空文件。
匹配策略 优先精确匹配；若 old_string 含换行，会尝试 LF/CRLF 互转匹配。
多匹配策略 匹配数大于 1 且 replace_all 非 true 时拒绝，要求更精确 old_string 或显式全量替换。
换行保真 替换文本按实际匹配到的 oldString 换行风格归一化：旧内容是 CRLF 时 new_string 转 CRLF；旧内容是 LF 时转 LF。
审批描述 describeApproval 先 prepareWriteTextFile，根据 create / replace all / unique edit 生成 summary，details 包含 target path 和 workdir。
能力声明 kind="write"、readOnly=false、destructive=true、concurrency="exclusive"。
注意：当前 tool.contract.test.ts 的相关测试已从“保留旧别名并保留 CRLF”改为“保留 CRLF”。如果仍需要向后兼容 path/search/replace/all，当前实现与测试意图已经不再覆盖这件事。

12. 设置 API 与配置持久化

Agent HTTP API

路由方法返回/输入实现

15. 边界与注意事项

事项当前事实影响
全局工具选择默认含义 空 tools 不是禁用，而是默认全部内置工具启用。 “Reset to default” 会写入空选择，恢复默认启用状态。
API 与 alias 运行时读取 selection 时会检查 id + aliases；但 updateBuiltinToolSelection 当前只接受内置工具 id。 手工配置 alias false 可能影响运行时；通过当前 HTTP 更新接口保存包含 alias 的 selection 会被视为 unknown builtin tool。
权限 fallback DEFAULT_ACTIONS 当前对所有 kind 都是 allow。 安全性主要依赖路径边界、critical 风险阻断、planning/side-chat 限制、工具自身 assessPermission。
full_access 模式 只改变 intent=ask 的结果；forceAsk、critical、deny、side-chat、外部路径写入等仍会阻断。 它不是无条件跳过安全策略。
provider-safe 名称 含点号的旧别名不暴露给模型。 例如 multi_tool_use.parallel 可作为 registry alias，但模型工具名是 multi_tool_use_parallel。
replace-text 兼容性 当前工作树移除了旧参数 alias。 外部调用方若仍发送 path/search/replace/all 会 schema 校验失败。
desktop Tools 页面边界 页面展示 inputSchema、capabilities、alias 数，但不展示实时 MCP/custom 工具。 它是“内置工具全局可用性”管理页，不是所有可调用工具总览。
